# 数据库架构优化方案

> **⚠️ 方案二已实施**: 当前系统已采用双表架构（`request_metadata` + `request_details_active`），方案二描述的架构已在生产环境中运行。本文档保留方案一作为参考对比。

> **导航**: [文档中心](../../README.md) | [当前设计](schema.md) | [归档机制](archive.md)

## 背景

> **注意**: 以下描述的 `request_records` 单表设计已被替换为双表架构（`request_metadata` + `request_details_active`），详见当前 [schema.md](schema.md)。

原 `request_records` 表存储所有请求的完整轨迹数据，随着数据量增长面临以下问题：

1. **查询性能下降**：按 `session_id` 或时间查询变慢
2. **存储空间压力**：每条记录包含多个 JSONB 大字段，体积较大
3. **数据管理困难**：历史数据难以清理，备份成本高

### 原表结构特点（已废弃）

> 以下为原 `request_records` 单表的特征，现已迁移为双表架构。

- 约 30 个字段，其中 6 个大型 JSONB 字段（`raw_request`, `raw_response`, `text_request`, `text_response`, `token_request`, `token_response`）
- 单条记录约 50KB（完整数据）
- 主要查询模式：按 `session_id` 查询、按时间范围查询、统计聚合

## 方案简述（对比参考）

> 以下只保留关键对比。完整的方案一/方案二设计细节已归档在 Git 历史中，需要时可查阅早期版本。

| 维度 | 方案一：活跃表 + 归档索引表 | 方案二：元数据表 + 详情表 (✅ 已实施) |
|------|---------------------------|-------------------------------------|
| 表数量 | 2 个（活跃 + 索引） | 2 个（元数据 + 详情） |
| 数据分离 | 活跃/历史完全分离 | 统计/详情分离 |
| 统计字段冗余 | 有（两个表各存一份） | 无（只在元数据表） |
| 清理效率 | 高（DROP 分区元数据操作） | 中（但通过 run_id 粒度 DELETE + 月分区 DETACH 缓解） |
| 历史统计查询 | 分别查两个表需应用层合并 | 只查元数据表（统一） |
| 回滚能力 | 困难（分区已删除） | 较易（元数据保留） |

**选择方案二的理由**：
- 跨时间统计查询是高频需求，方案一只靠元数据无法满足
- 避免统计字段冗余，降低存储和一致性问题
- 通过按月分区详情表 + run_id 粒度归档，清理效率损失可控

## 当前架构（方案二实施后）

当前生产环境已采用方案二，核心设计如下：

```
┌────────────────────────────────────────────────────────────┐
│                    request_metadata                         │
│              (元数据表，长期保留)                            │
│   unique_id, session_id, model, tokens, archive_location   │
└────────────────────────────────────────────────────────────┘
         │ archive_location IS NULL      │ NOT NULL
         ▼                               ▼
┌─────────────────────────┐    ┌─────────────────────────┐
│  request_details_active │    │    外部存储              │
│  (按月分区，近期大字段)   │    │  {run_id}/{session}.   │
│  messages, raw_* 等      │    │  jsonl.gz              │
└─────────────────────────┘    └─────────────────────────┘
```

完整的表结构、索引、数据类定义详见 [数据库设计](schema.md)。

归档机制详见 [归档机制](archive.md)。
