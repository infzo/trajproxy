## 概述

TrajProxy 有一个 **Grafana Dashboard**：**TrajProxy Overview**，共 **43 个面板**，按功能从上到下分为 7 个区域（Row），阅读顺序即巡检顺序：

```
Row 0  全局总览       → 一眼看全：谁在用、用了多少、快不快
Row 1  流量概览       → 三类业务的健康红绿灯
Row 2a 流量分布       → 各作业请求量、结果分类、重试、模式占比
Row 2b 耗时与序列     → 端到端延迟、推理调用耗时、首 Token、序列长度
Row 2c 吞吐量分析     → QPM、活跃作业数、Prefill/Decode 吞吐
Row 2d 失败分析       → 失败数量、错误细分
Row 3  轨迹请求       → 深入排查轨迹查询的性能和错误
Row 4  系统资源       → 底层：内存、CPU、连接池、存活状态
```

---

## 顶栏筛选器

Overview Dashboard 顶部有 **6 个下拉框**，可缩小数据范围，所有面板联动：

| 筛选器 | 作用 | 使用场景 |
|--------|------|----------|
| **数据源** | 选择 Prometheus 实例 | 多环境时切换 |
| **节点** | 按 TrajProxy 节点筛选 | 排查某个特定节点的问题 |
| **模型** | 按推理模型名筛选 | 排查某个模型的请求表现 |
| **Run ID** | 按作业标识筛选 | 排查某个具体任务或实验的数据 |
| **实例** | 按 Prometheus 采集目标筛选 | 区分不同进程实例 |
| **分位数** | 选择 P50 / P95 / P99 | 观察不同分位下的延迟分布：P50 看大部分用户体验，P95 看尾部延迟，P99 看极端情况 |

## Dashboard 1: TrajProxy Overview

---

### Row 0 — 全局总览

> **回答的核心问题**: 现在系统有多少请求？都是什么类型？快不快？

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **全站请求类型占比 (1h)** | 过去 1 小时各类型请求的占比饼图 | 快速判断流量结构是否正常。推理请求应占主导，如果轨迹查询突然变多可能有异常 |
| **全站请求数 (5min) — 按请求类型** | 每 5 分钟各类型请求的数量变化曲线 | 观察流量是否在平稳波动。突然出现尖峰或骤降都需要关注 |
| **全站请求数 (5min) — 按节点** | 每 5 分钟各节点收到的请求数曲线 | 多节点部署时看负载均衡是否均匀。某节点明显偏高说明分发不均 |
| **全站请求耗时 — 按请求类型** | 各请求类型的响应耗时趋势 | 对比不同类型哪个更慢。标题会显示当前选的分位数（如 P95） |
| **模型生命周期事件** | 模型加载 / 卸载的次数 | 频繁出现说明模型在不断重载。偶尔出现是正常的按需加载 |

---

### Row 1 — 流量概览

> **回答的核心问题**: 三类业务（推理、模型管理、轨迹查询）各自的「健康度」如何？

三个面板并排，结构相同，每个都有三项指标：

| 面板 | 适用对象 |
|------|----------|
| **推理请求概览** | 核心推理服务（chat/completions） |
| **模型请求概览** | 模型注册 / 删除 / 列表 API |
| **轨迹请求概览** | 轨迹查询（列表 + 详情）API |

**每个面板都展示**：

- **请求数** — 最近 5 分钟的请求量，显示趋势迷你图
- **成功率** — 颜色是红绿灯
  - 绿色 ≥ 99%：一切正常
  - 黄色 90~99%：需要关注
  - 红色 < 90%：需要立即排查
- **平均耗时** — 颜色也是红绿灯（推理请求：< 10s 绿、10~30s 黄、> 30s 红；管理和轨迹请求：< 1s 绿、1~5s 黄、> 5s 红）

**巡检建议**: 这是最优先看的一行。三个面板都是绿灯就不用往下看；出现黄色或红色就到对应 Row（推理→Row 2，轨迹→Row 3）深入排查。

### Row 2 — 推理请求

> **回答的核心问题**: 推理请求出了什么错？慢在哪里？

#### 第一排：流量分布

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **推理请求数 — 按运行标识** | 每个作业（Run ID）发来了多少请求 | 看哪个作业在用、流量多大 |
| **推理请求数 — 按结果** | 每 5 分钟的成功/失败请求数，按结果分类显示 | **最重要的错误面板**。绿色是成功，其他颜色都是问题（见下方颜色说明） |
| **推理 Pipeline 各阶段耗时** | 一次推理请求在各阶段花了多少时间 | 哪条线高就是哪个阶段在拖后腿：推理阶段高 → 下游慢；转换阶段高 → 模板渲染慢；存储阶段高 → 数据库慢 |
| **推理流式与非流式占比 (1h)** | 流式 vs 非流式请求饼图 | 流量模式评估。流式占比高时更应关注 TTFT 指标 |
| **DIRECT vs TITO 模式占比 (1h)** | 直接转发 vs Token 模式饼图 | 评估两种处理管道的使用比例 |

**结果分类颜色说明**：

| 颜色 | 含义 | 遇到怎么办 |
|------|------|-----------|
| 绿色 | 请求成功 | 正常 |
| 浅绿色 | 流式传输部分完成（如客户端中途断开） | 检查客户端网络 |
| 橙色 | 客户端请求不合法（4xx） | 检查请求方发送的参数格式 |
| 深红色 | Proxy 内部错误（5xx） | 检查 TrajProxy 日志 |
| 紫色 | 请求被限流（429） | 并发太高被拒绝，考虑调大并发配置或减少请求量 |
| 蓝色 | 推理服务异常（连接失败/HTTP 错误） | 检查下游推理服务是否正常运行 |
| 红色 | 推理服务超时 | 推理服务处理太慢，或增大超时时间 |

#### 第二排：吞吐量分析

> **回答的核心问题**: 各运行标识的请求速率和 Token 吞吐是多少？哪个作业消耗最多？

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **每分钟请求数 QPM - 按运行标识** | 各 run_id 每分钟推理请求速率（QPM），实时反映请求压力 | QPM 越高该作业产生的负载越大。突然飙升可能是上游突发流量或重试风暴 |
| **活跃运行标识数量 (5min)** | 过去 5 分钟内有请求活动的 run_id 总数 | 反映同时运行的作业数量。数量越多说明系统并行度越高，也意味着单个作业的可用资源更少 |
| **Prefill 吞吐量 (Tokens/s) - 按运行标识** | 各 run_id 每秒 Prompt Token 消耗速率（Prefill 阶段） | 衡量输入侧处理负载。某 run_id 持续高位说明其请求的输入序列较长或并发量大 |
| **Decode 吞吐量 (Tokens/s) - 按运行标识** | 各 run_id 每秒 Completion Token 产出速率（Decode 阶段） | 衡量输出侧生成速度。Decode 吞吐直接影响用户感知的响应速度，越高说明生成越快 |
| **推理内部重试次数 — 按节点** | 各节点每 5 分钟 Proxy 自动重试次数 | 柱子越高说明推理服务越不稳定，经常返回 5xx 被自动重试。按节点分布可定位问题节点 |

#### 第三排：耗时与序列

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **推理请求端到端耗时 — 按运行标识** | 从收到请求到返回响应的总耗时 | 最能反映用户体验的面板。可以和下面的「推理服务调用耗时」对比，判断瓶颈在哪 |
| **推理服务调用耗时 — 按运行标识** | 调用下游推理服务器的纯网络耗时 | 如果比「端到端耗时」小很多，说明 Proxy 自身的转换/存储花了不少时间 |
| **推理首 Token 耗时 (TTFT)** | 流式请求中，从发出请求到收到第一个字的时间 | 用户感受「响应快不快」的关键指标。越高说明推理服务 prefill 越慢 |
| **推理序列长度（输入+输出）** | 每次请求的 Token 总量（输入 + 输出） | 评估请求的规模大小。越大越慢是正常趋势 |
| **TITO 缓存命中率 — 按运行标识** | 前缀缓存的命中率（仅 TITO 模式有数据） | 越高越好。低于 50% 说明多轮对话共享的前缀较少 |

#### 第四排：失败分析

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **推理请求失败数量 — 按节点** | 各节点的失败请求数 + DB 写入失败数 | 定位是哪个节点出了问题。左轴是请求失败，右轴是轨迹写入失败 |
| **推理请求失败数量 — 按运行标识** | 各作业的失败请求数 + DB 写入失败数 | 定位是哪个作业出了问题 |
| **推理请求错误细分** | 把错误按类型拆开显示（堆叠柱状图） | 在顶栏选一个 Run ID，看这个作业的错误都是什么类型 |

**失败分析排查顺序**：先看「按节点」→ 缩小到问题节点 → 再看「按运行标识」→ 缩小到问题作业 → 最后看「错误细分」→ 确认错误类型

### Row 3 — 轨迹请求

> **回答的核心问题**: 轨迹查询慢不慢？数据量大不大？有没有出错？

#### 第一排：请求量与耗时

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **轨迹详情请求数 — 按运行标识** | 各作业的轨迹查询次数 | 看哪个作业的查询比较频繁 |
| **轨迹详情请求耗时 — 按运行标识** | 轨迹查询的响应时间 | 如果很高，看下面两个面板判断是不是数据量太大 |
| **轨迹平均对话数量** | 每次查询返回的平均对话记录条数 | 条数越多查询越慢是正常趋势。如果特别多，考虑用 `fields` 参数过滤返回字段 |
| **轨迹请求平均大小** | 每次查询返回的数据量（自动显示 KB/MB/GB） | 超过 10MB 可能导致超时。同样可以通过 `fields` 参数减小体积 |

#### 第二排：失败分析

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **轨迹请求失败数量 — 按节点** | 各节点的轨迹查询失败数 | 定位问题节点 |
| **轨迹请求失败数量 — 按运行标识** | 各作业的轨迹查询失败数 | 定位问题作业 |
| **轨迹请求错误细分** | 把轨迹错误按类型拆开 | 红色=数据库异常，橙色=查询超时，黄色=序列化超时，紫色=限流，灰色=其他 |

---

### Row 4 — 系统资源

> **回答的核心问题**: 服务还活着吗？资源够用吗？

| 面板 | 作用 | 怎么看 |
|------|------|--------|
| **Worker 存活状态** | 各实例是否在线（UP/DOWN） | 绿色 UP = 正常，红色 DOWN = 实例挂了，**立即处理** |
| **Worker 内存 (RSS)** | 各进程的内存占用趋势 | 持续上涨需要关注是否有内存泄漏。超过 2GB 红色线就需要注意 |
| **CPU 使用率** | 各进程的 CPU 占用百分比 | 长期高于 80% 需要排查是否有阻塞操作 |
| **打开文件描述符** | 当前打开的连接数 | 一直涨不停可能是连接泄漏。接近 65535 就危险了 |
| **DB 连接池使用率** | 数据库连接的使用数量 | 接近连接池上限会导致请求排队甚至超时 |

## 告警规则

系统配置了 7 条自动告警，触发时会通知告警接收端：

| 告警 | 谁触发的 | 什么时候触发 | 级别 |
|------|----------|-------------|------|
| **WorkerDown** | Worker 存活状态 | 任一实例不可达超过 1 分钟 | **critical** — 立即处理 |
| **InferErrors** | 推理错误率 | 某模型的推理错误速率过高超过 3 分钟 | **critical** — 立即处理 |
| **HighErrorRate** | 请求错误率 | 某节点的错误率 > 5% 持续 5 分钟 | warning — 关注处理 |
| **HighLatency** | 请求延迟 | 某节点的 P95 延迟 > 60 秒持续 5 分钟 | warning — 关注处理 |
| **InferRetriesHigh** | 推理重试 | 某模型的重试频率过高持续 5 分钟 | warning — 关注处理 |
| **DBPoolSaturation** | DB 连接池 | 连接池使用率 > 85% 持续 5 分钟 | warning — 关注处理 |
| **WorkerHighMemory** | Worker 内存 | 某实例内存 > 2GB 持续 10 分钟 | warning — 关注处理 |

**级别说明**:
- **critical** — 服务已挂或核心功能不可用，必须立即响应
- **warning** — 性能下降或资源预警，需要关注处理

## 常见场景：我该看哪个面板？

### 场景 1：推理请求报错了

1. 看 **Row 1 推理请求概览** → 成功率是不是黄了/红了
2. 看 **Row 2 推理请求数 — 按结果** → 是哪种颜色的错误在升高
3. 顶部下拉框选对应的 Run ID → 看 **推理请求错误细分** → 确认具体错误类型
4. 如果是推理服务问题 → 看 **推理内部重试次数** 确认下游是否稳定

### 场景 2：推理请求变慢了

1. 看 **Row 2 推理请求端到端耗时** → 确认是否真的变慢
2. 看 **推理 Pipeline 各阶段耗时** → 哪条线高就是哪在拖后腿
3. 对比 **端到端耗时** 和 **推理服务调用耗时** → 差距大说明 Proxy 自身处理慢
4. 流式场景看 **TTFT** → 首字时间反映用户等待感

### 场景 3：轨迹查询很慢或报错

1. 看 **Row 1 轨迹请求概览** → 确认耗时是否超标
2. 看 **Row 3 轨迹请求平均大小** 和 **平均对话数量** → 数据量是否太大
3. 看 **轨迹请求错误细分** → 红色是数据库问题、橙色/黄色是超时

### 场景 4：想知道系统扛不扛得住

1. 看 **Row 4 Worker 存活状态** → 确认所有实例都活着
2. 看 **CPU 使用率** 和 **Worker 内存** → 资源水位
3. 看 **DB 连接池使用率** → 离上限还有多远
4. 看 **Row 0 全站请求数 — 按节点** → 负载是否均衡

## 附录：Dashboard 构建机制

Overview Dashboard 采用**分片构建**模式，源文件按 Row 拆分维护，通过脚本组装为完整的 dashboard JSON：

```
dockers/observability/configs/grafana/
├── dashboard-src/                 # 源文件（不会被 Grafana 扫描）
│   ├── dashboard.json             # 模板：元数据 + 变量定义
│   ├── row-0-overview.json        # Row 0 面板
│   ├── row-1-traffic.json         # Row 1 面板
│   ├── row-2-inference.json       # Row 2 面板
│   ├── row-2b-cache.json          # 缓存面板
│   ├── row-3-trajectory.json      # Row 3 面板
│   ├── row-4-system.json          # Row 4 面板
│   └── build_dashboard.py         # 组装脚本
└── dashboards/                    # 构建产物目录（Grafana 加载这里）
    └── trajproxy-overview.json    # 最终组装结果
```

**构建命令**：
```bash
make dashboard
# 或
python3 dockers/observability/configs/grafana/dashboard-src/build_dashboard.py
```

修改任何面板片段后需重新构建。
